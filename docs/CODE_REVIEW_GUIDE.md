# 🕵️‍♂️ คู่มือการทำ Code Review (สำหรับทีมและผู้เริ่มต้น)

ยินดีด้วยครับที่คุณและทีมก้าวเข้าสู่การเขียนโปรแกรมและการทำ Code Review เป็นครั้งแรก! 
การทำ Code Review ไม่ใช่การจับผิด แต่เป็นการ **"ทำความเข้าใจโค้ดร่วมกัน"** และ **"หาช่องโหว่เพื่อป้องกันระบบพังในอนาคต"**

เอกสารฉบับนี้ถูกเขียนขึ้นเพื่อเป็น "แผนที่" ให้คนในทีมรู้ว่าควรเริ่มเปิดอ่านและตรวจโค้ดไฟล์ไหนก่อน-หลัง และควรจับตามองจุดไหนเป็นพิเศษสำหรับโปรเจค AutoMerge นี้ครับ

## 🗺️ ลำดับการเปิดอ่านและตรวจโค้ด (Order of Review)

เพื่อให้สมองเรียบเรียงได้ง่ายที่สุด แนะนำให้อ่านจากรากฐานของระบบ ไปจนถึงหน้าบ้านครับ

### 🟢 ลำดับที่ 1: รากฐานและฐานข้อมูล (Infrastructure)
เริ่มตรวจจากจุดที่เป็นโครงสร้างพื้นฐานที่สุด ถ้ารากฐานผิด ส่วนอื่นจะผิดตามหมด
1. **`config/settings.py`**
   - **สิ่งที่ต้องตรวจ:** เช็คว่าค่า Default ต่างๆ เช่นชื่อ Database (`DATABASE_URL`) ถูกต้องไหม และมีการดึงข้อมูลจาก Environment Variable (`.env`) อย่างปลอดภัยหรือไม่
2. **`storage/models.py`**
   - **สิ่งที่ต้องตรวจ:** ดูตาราง `RawFile` และ `RawData` ว่ามีคอลัมน์ไหนพิมพ์ชื่อผิดไหม ตรวจสอบว่าคอลัมน์ `file_hash` และ `row_hash` ถูกสร้างไว้ป้องกันข้อมูลซ้ำจริงๆ หรือไม่
3. **`storage/database.py`**
   - **สิ่งที่ต้องตรวจ:** ไฟล์นี้สำคัญมาก! ตรวจสอบว่าเรามีการตั้งค่า **Database Connection Pool** (เช่น `pool_size=20`, `max_overflow=30`) แล้วหรือยัง เพื่อยืนยันว่าระบบจะไม่ล่มเวลามีคนอัปโหลดไฟล์พร้อมกันเยอะๆ

### 🟡 ลำดับที่ 2: เส้นทางการรับข้อมูล (API Endpoint)
ส่วนนี้คือประตูด่านแรกที่รับไฟล์จากผู้ใช้งาน (Frontend)
4. **`api/main.py` (โฟกัสที่ฟังก์ชัน `upload_file`)**
   - **สิ่งที่ต้องตรวจ:** 
     - เช็คว่า API มีการดักจับไฟล์ซ้ำ (Duplicate) ด้วยการเช็ค Hash และตีกลับด้วยสถานะ `409 Conflict` หรือไม่
     - เช็คว่าไม่ได้เอาไฟล์โหลดเข้า RAM ตรงๆ แต่อนุญาตให้เซฟลงโฟลเดอร์ Temp ก่อน
     - เช็คว่า API ปล่อยงานให้ `BackgroundTasks` ทำต่อ แทนที่จะประมวลผลเองจนค้าง

### 🟠 ลำดับที่ 3: ระบบประมวลผลหลังบ้าน (Background Worker)
นี่คือส่วนที่ทำงานหนักที่สุดและเสี่ยงต่อการพังที่สุด
5. **`api/main.py` (โฟกัสที่ฟังก์ชัน `process_uploaded_file`)**
   - **สิ่งที่ต้องตรวจ (สำคัญที่สุด):**
     - มีคำสั่ง `db.rollback()` ในบล็อก `except` หรือไม่? (ถ้าเกิด Error ระหว่างทาง ฐานข้อมูลต้องย้อนกลับไปจุดเริ่มต้น ห้ามพังครึ่งๆ กลางๆ)
     - มีการลบไฟล์ Temp ทิ้งในบล็อก `finally:` หรือไม่? (ป้องกันฮาร์ดดิสก์เต็ม)
     - มีการหั่นข้อมูลเป็นก้อนๆ (Chunk) ระหว่างส่งเข้า Database ไหม?

6. **`ingestion/file_parser.py`**
   - **สิ่งที่ต้องตรวจ:** ตรวจสอบกระบวนการแกะไฟล์ CSV/Excel ดูว่าใช้ฟังก์ชันของ Pandas (`read_csv`, `read_excel`) ถูกต้อง และดักจับ Error กรณีผู้ใช้อัปโหลดไฟล์ขยะมาหรือไม่

### 🔵 ลำดับที่ 4: ส่วนแสดงผลและการทดสอบ (Frontend & Testing)
7. **`testing/generate_test_files.py` และ `testing/stress_upload.py`**
   - **สิ่งที่ต้องตรวจ:** ทบทวนว่าเครื่องมือสร้างข้อมูลปลอมทำงานได้จริง และไม่ได้บังเอิญไปแตะต้องฐานข้อมูลของจริง
8. **`frontend/index.html`**
   - **สิ่งที่ต้องตรวจ:** ดูว่า JavaScript ฝั่งหน้าบ้านส่งไฟล์แบบ Multiple เข้าหา API ได้อย่างถูกต้อง และแสดงผล Error (เช่น 409 ไฟล์ซ้ำ) ให้ User เข้าใจได้ง่ายๆ

## 🎯 Checklist เบื้องต้นสำหรับคนรีวิวโค้ดมือใหม่
เวลาเปิดไฟล์ขึ้นมา ไม่ต้องพยายามเข้าใจคำสั่งแปลกๆ ทุกตัวอักษร แต่ให้ถามตัวเองด้วยคำถาม 3 ข้อนี้ครับ:
- [ ] **"ถ้ามีคนอัปโหลดไฟล์ขนาดใหญ่ 1GB เข้ามา โค้ดบรรทัดนี้จะพังไหม?"** (เราป้องกันด้วย Temp File แล้ว)
- [ ] **"ถ้าคนอัปโหลดไฟล์พร้อมกัน 100 คน ฐานข้อมูลจะล่มไหม?"** (เราป้องกันด้วย DB Pool และ BackgroundTasks แล้ว)
- [ ] **"ถ้าไฟดับ หรือโค้ด Error กลางคัน ข้อมูลในระบบจะเละเทะไหม?"** (เราป้องกันด้วย `db.rollback()` แล้ว)
